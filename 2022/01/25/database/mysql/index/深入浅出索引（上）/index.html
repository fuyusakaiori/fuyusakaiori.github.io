深入浅出索引（上）

前言

你如果发现某个 SQL 语句执行过慢的时候，可能通常会考虑添加索引来加快执行速度，再者如果发现磁盘占用率过高的时候，可能就是因为索引添加过多导致占用空间过多。在学习或者工作的过程中会遇见很多涉及索引的情况，那么索引到底是什么？索引应该在什么时候使用，是在表刚建立完成的时候添加？还是在业务的执行过程中添加？索引又是如何加快查询的速度的呢？我打算用三篇笔记将索引的常见知识给讲述清楚

先不从定义的角度去考虑索引到底是什么东西，而是先思考下如何才能够加快在大量数据中的查询速度。最直接的例子就是生活中常用的字典，你在查询某个字的时候肯定不是直接去字典中大海捞针，而是会首先根据字典的目录去确定这个字在哪一页。

数据库也是类似的道理，既然大量数据存储在磁盘的数据页中，显然是不可能全盘扫描去查询的，所以也会设计相应的目录去加速查询的过程，那么这个目录对应的就是数据库中的索引

再从字典的角度去考虑，每次通过目录查询字的时候通常会有三种查询方式：声母查询、偏旁查询、笔画查询。如果采用声母查询，那么首先就要先翻到需要查询的字的拼音首字母所在的索引页，然后再根据字的读音继续查询，最终就可以找到相应的字所在的页。如果采用偏旁查询，那么通常会配合笔画查询来进行索引。

数据库的 B+ 树索引模型也是完全相同的道理，声母查询几乎是容易使用的查询方式，和聚簇索引类似（主键），偏旁查询和笔画查询则通常没有办法唯一确定，和普通索引类似（二级索引），而两者的联合使用就是所谓的联合索引。

根据上述的例子，这样就可以总结出索引到底是什么东西了

索引就是采用各种数据结构实现的用于加快查询速度的一种目录

索引模型

概述

索引（目录）的实现方式有很多种，许多常见的数据结构都可以提高查询的速度，比如哈希表、有序数组、二叉搜索树及其衍生、跳表等等。在介绍常见的索引模型之前，会基础概念进行说明，避免在之后重复解释

1、查询方式

  • 等值查询：查询得到的结果通常都是唯一的
  • 范围查询：查询得到的结果通常是集合

2、记录和数据页

索引模型中提及的结点、元素或者说键值对名词等等，在物理存储结构中实际对应是数据页而不是记录，数据页中存储的才是我们需要的记录，不过在介绍索引模型的时候不做详细的区分，需要了解什么是数据页的话可以查看之前写的笔记

​

有序数组

  • 基本思想：

    • 将所有记录都进行有序化处理（递增或者递减）
    • 每次查询的时候直接使用二分查找法（时间复杂度为 O(logN)）

  • 特点：

    • 有序数组支持范围查询和等值查询
      • 每次只需要找到第一个满足条件的记录，之后就可以直接向后遍历找到其他满足条件的记录
    • 仅看有序数组的查询效率的话，是所有索引模型中查询速度最快的
    • 有序数组每次插入和删除记录就非常低效，需要移动后面所有记录，仅适合静态存储引擎

注：静态存储引擎这里指的就是通查不怎么发生变化的静态表（系统配置表）

哈希表

  • 基本思想
    • 每次使用相应的哈希函数将 Key 值计算出对应的哈希值，然后将 Value 放在哈希值对应的位置中
    • 出现哈希冲突的时候通常采用拉链法，也就是将具有相同哈希值的记录组织成单链表的形式
    • 每次查询的时候根据查询的 Key 值计算出对应的哈希值，然后遍历相应的位置上的链表找到符合条件的记录（时间复杂度O(K)）
  • 特点：
    • 哈希表仅支持等值查询不支持范围查询：
      • 因为哈希表是无序的，所以每次查询都需要先计算出对应的哈希值然后遍历链表
      • 没有办法像有序数组一样在找到第一个满足的条件的记录后直接遍历
    • 哈希表每次插入和删除记录是非常容易的，只需要修改链表中的指针就可以（时间复杂度O(1)）
    • 哈希表仅适用于 Memcached 等缓存或者 NoSQL 存储引擎（注：Redis 中采用的是跳表作为索引）

二叉搜索树

1、二叉搜索树索引

  • 基本思想

    • 保证父结点的左子结点始终小于自己，右子结点始终大于自己
    • 每次查询的时候，如果查询的值小于当前结点那么向左遍历，如果查询的值大于当前结点那么向右遍历（时间复杂度O(logN)）

  • 特点：

    • 二叉搜索树也只支持等值查询不支持范围查询
      • 每次查询到满足的条件的记录之后可以从左右子树开始深度遍历，但是只能够找到部分比自己大的记录
      • 因为可能该记录的父结点的其余子树也存在符合条件的记录，所以只能够通过回溯再次去遍历可能符合条件的记录
    • 二叉搜索树的查询速度最好才是 O(logN)，最差由于可能退化成链表所以时间复杂度为 O(N)

​

2、二叉平衡树

此前提到的二叉搜索树存在可能退化成单链表的情况，从而会严重影响查询数据的速度，那么是否可以避免二叉搜索树退化成单链表呢？

  • 基本思想：
    • 二叉平衡树就可以通过左旋和右旋的操作不断调整二叉树的高度，从而避免退化成单链表
    • 二叉平衡树的时间复杂度通常都能够保证在 O(logN)
  • 特点：（这里仅以 AVL 树和红黑树作为代表，其余衍生的二叉平衡树不做解释）
    • 如果采用 AVL 树实现平衡，调整高度的过程会涉及到大量的左旋和右旋从而造成性能不佳的情况
    • 如果采用红黑树实现平衡，虽然可以改善 AVL 树左旋和右旋的性能问题，但是仍然存在非常严重的问题

每次查询都会去磁盘中查找符合要求的记录，然后将符合条件的记录所在的数据页加载到内存中，在内存中遍历加载进入的数据页从而获取到符合条件的记录，这就意味着每次查询都可能产生读写磁盘的开销

而二叉树每个结点仅存储两个值，每个结点又对应一个索引页或者数据页，这也就意味着每个数据页仅能够存储两行记录。如果在数据量特别大的情况下就会导致二叉树的结点非常多且深度很大

最终就会导致磁盘的随机读写过于消耗时间，从而导致查询效率低下

注意事项：

1. 红黑树存在的问题是整个二叉搜索树都存在的问题，并不是说其余二叉搜索树作为索引就没有这个问题

2. 二叉树确实会造成磁盘读写开销增大，但是到底是因为二叉树深度过高还是二叉树结点过多并不是特别清楚，但是我更倾向于后者

3. 这里提到的磁盘读写开销都是基于机械硬盘的，固态硬盘下可能开销会小一点

3、多路查找树

此前二叉平衡树和二叉搜索树因为单个结点存储的数据是有限的，所以没有办法解决树的结点过多和高度过大的情况，那么是否有其他的树可以让每个结点存储多个值呢？

  • 基本思想：
    • 多路查找树可以允许每个父结点拥有多个子结点，并且依然遵循二叉搜索树从小到大的基本特性
    • 多路查找树还会通过分裂和合并的行为避免树结构退化成链表结构，时间复杂度不清楚
  • 类型：① B 树 ② B+ 树 ③ B* 树

此前已经提到了为什么不采用二叉搜索树或者二叉平衡树这两种作为索引模型，那么为什么多路查找树就可以作为索引模型呢？

多路查找树允许每个父结点存储多个子结点，那么就意味着每个数据页此时就可以存储多行记录。这样即使在数据量特别大的情况下就不会产生大量的结点且高度也是可以接收的。最终就可以减少磁盘随机读写的时间，从而提升查询效率

那么为什么数据库中从来都不采用 B 树作为索引模型而是采用 B+ 树呢？

B+ 树的特点就在于非叶子结点不存储数据仅存储指向叶子结点的指针域，这样就能够让非叶子结点存储更多的指针，可以让树的结点数量和高度进一步减小，所以效率会比 B树要高，因为B树每个结点都存储数据。此外，B+树的底层还会采用双向链表将叶子结点串联起来，从而支持范围查询
