CAS 机制

为什么需要 CAS 机制？

[回忆确保线程安全的方式]{.label .danger}

[此前了解到的确保线程安全的方式，都是采用一致的锁定协议来协调对共享变量的访问，也就是对共享变量上锁]{.pink}

  • synchronized
  • ReentrantLock
  • ReentrantReadWriteLock

[可以确保无论哪个线程持有共享变量的锁，都可以采用独占式的方式来访问共享变量，并且对共享变量的任何修改对随后访问这个共享变量的线程都是可见的]{.pink}

[锁定协议的性能分析]{.label .primary}

[存在多个线程同时竞争共享变量时，锁的独占性会让其余没能成功获得锁的线程全部进入阻塞队列中等待，而 线程进入阻塞队列需要发生上下文切换，内核级线程的上下文切换需要操作系统从用户态陷入内核态，这个过程会造成非常大的开销]{.purple}

[上下文切换造成的开销有好几种变体方式，比如 ① 线程持有锁的时间非常短暂，而其余线程在这短暂的时间里突然请求锁的使用权，那么就会出现线程才刚刚因为上下文切换陷入阻塞，又因为锁的释放而被唤醒，这就会造成很高的开销 ② 线程的竞争不够激烈的情况下，这里的不够激烈意思是持有锁的线程在执行过程中没有太多的线程来竞争锁的使用，那么这时候依然让线程阻塞挂起就会]{.purple}

[如果在临界区中 对共享变量的操作较少，也就是持有锁的时间非常短，那么就会出现频繁竞争锁、释放锁的过程，也就是 会频繁出现上下文切换 的过程，会造成非常大的开销]{.purple}

[为了减少锁定协议带来的开销，JDK 6 之后添加非常多对 synchronized 锁的优化措施，从而 减少在没有线程竞争下的开销，但是在 中低程度的竞争下开销依然无法避免]{.purple}

  • 轻量级锁、偏向锁（批量重偏向、批量撤销）
  • 自旋锁、锁消除、锁同步、锁粗化

[锁定协议存在的其他问题]{.label .warning}

  • 阻塞式调用

    [持有锁的线程调用阻塞式的方法，导致线程长时间持有锁但却什么也不做，其余想要获得锁的线程依然处于阻塞，造成系统的整体执行效率降低]{.orange}

  • 优先级反转

    [优先级较低的线程抢到锁的使用权，优先级高的线程必须等待优先级较低的线程释放锁才能够执行，如果频繁出现这种情况，那么优先级高的线程实质上已经变成了低优先级线程，这就是优先级反转问题]{.orange}

  • 产生死锁、饥饿、活跃性问题（这部分问题会单独讲述）

:::info

上述的阻塞式调用和死锁问题实际都有相应的方式解决，但是无法避免这些问题是实际存在的

:::

[volatile 关键字]{.label .info}

[volatile 是轻量级的同步机制，可以确保可见性和有序性，不会让线程发生上下文切换，但是它是无法保证线程操作的原子性的，这个缺陷就导致 volatile 无法替代锁的存在]{.blue}

[总结]{.label}

锁定协议在中低程度的竞争下的开销和工作开销的比值相对较大，并且难以通过优化措施去避免上下文切换的开销；其次锁定协议还会造成非常经典（严重）的死锁、饥饿等问题，以及优先级反转问题。所以我们开始考虑是否存在一种方式能够避免这些开销的同时还能够确保原子性，这个问题的答案就是 CAS 机制

CAS 机制存在的缺陷

ABA 问题

[什么是 ABA 问题]{.label .primary}

  • 定义：[变量在当前线程不知情的情况下被其余线程修改后又变回原本的值，而当前线程依旧认为没有发生变化]{.purple}
    • 也就是说存在某个线程想要利用 CAS 更新将共享变量的值从 A 更新为 B
    • 但是中途被其余线程从 B 更新为 C，又从 C 更新为 B，此时这个线程依旧会认为共享变量的值没有变化
  • 核心：[共享变量的状态值产生环形转换]{.red}
  • 细节：[虽然 ABA 问题存在线程安全隐患，但是通常都不会造成很严重的问题]{.pink}

[如何解决 ABA 问题]{.label .success}

  • 核心：[避免共享变量的状态值产生环形转换]{.red}
  • 方式：[版本号机制]{.red}
    • 原理：
      • [每次更新会同时更新共享变量和版本号两个值，共享变量可以产生环形转换，但是版本号始终递增]{.pink}
      • [每次更新的时候不仅检查共享变量是否符合期望值，同时还会检查版本号是否符合期望值]{.pink}
    • 工具类：[AtomicStampedReference、AtomicMarkableReference]{.pink}

暂停线程

Sleep 方法

  • 作用：[让正在运行的线程进入睡眠状态，虚拟机/处理器在此期间不会调度该线程]{.red}

  • 使用：

    • [JDK 1.0 默认提供睡眠方法：默认时间单位是毫秒（ms）]{.blue}

      • 核心方法：Thread.sleep()

      • 代码演示

    • [JDK 1.5 提供的新的睡眠方法：单位可以根据需要使用]{.blue}

      • 核心方法：Time.Unit.SECONDS.sleep()

      • 代码演示

  • 特点：

    • [Sleep 方法会让线程主动交出处理器的使用权，也就是放弃自己的时间片]{.red}
    • [Sleep 方法不会释放锁的所有权，也就是持有锁的线程睡眠时，别的线程依旧无法获取锁]{.red}
    • [Sleep 方法会让线程从 RUNNABLE 转为 TIMED_WAITING 状态]{.red}
    • [正在休眠的线程是可以被中断的，线程被中断之后会默认抛出异常，并且将中断标志重新置为 false]{.pink}
    • 休眠结束的线程需要重新和其他线程竞争处理器的使用权

中断方法和锁相关内容之后会详细说明

Yield 方法

  • 作用：[提示正在运行的线程主动放弃处理器的使用权，从而让其他线程执行]{.red}

  • 使用：（这个方法的效果不是很明显，不过也有办法测试）

    • 测试方式

      • 第一条线程不做任何礼让，持续执行增加数量的操作；第二条线程会边进行礼让边执行增加数量的操作
      • 观察到的结果应该是第一条线程输出语句更多，第二条线程输出语句更少，因为进行了礼让

      :::warning

      ① 线程上下文切换和线程礼让都会导致运行的线程发生改变

      ② 所以只用一次 Thread.Yield 是没有办法判断究竟是线程切换还是线程礼让造成的

      :::

    • 测试代码

  • 特点：

    • [正在运行的线程可能放弃也可能拒绝放弃自己对于处理器的使用权，完全取决于线程自己]{.red}
      • Yield 方法的效果和线程优先级的效果类似，都是没有办法强制控制线程的
    • [Yield 方法会让线程从 RUNNABLE（running）转换为 RUNNABLE（ready）状态]{.red}
    • [Yield 方法也不会释放锁的所有权]{.red}

Join 方法

  • 作用：[调用线程阻塞式等待正在运行的线程结束]{.red}

  • 使用：

    • 核心方法：thread.join()（这里的 thread 是等待的线程对象而不是调用线程或者 Thread 类）

    • [无限期阻塞等待：调用线程会等待线程运行结束才开始运行]{.pink}

    • 有限期阻塞等待

      • [如果等待的线程提前结束，那么调用线程也会提前结束]{.red}

      • [如果调用线程已经等待了设定的时间，等待的线程依然没有结束，那么调用线程就会不再等待]{.red}

  • 特点：

    • [Join 方法的本质就是 Wait 方法]{.red}

    • [调用线程 从 RUNNABLE 转换为 WAITING 状态]{.red}

    • [调用线程到的阻塞等待过程是可以被其余线程中断的]{.red}

    • [调用线程阻塞等待多个线程时，等待的时间取决执行时间最长的那个线程]{.pink}

      1. 调用线程等待 Thread-1、Thread-2 两个线程，Thread-1 执行 1秒，Thread-2 执行 2秒
      2. 调用线程阻塞式等待这两个线程，那么调用线程最终只会等待 2 秒，而不是 3 秒
      3. 多核处理的情况下线程基本都是并行，所以显然只需要等待 2 秒

Interrupt 方法

  • 作用：[设置需要中断的线程的中断标志位 true，线程根据中断标志 自行决定 之后的行为]{.red}

  • 使用：

    • 核心方法：thread.interrupt()

    • 代码演示

  • 特点：

    • 中断标志默认设置为 false
    • [Interrupt 方法可以中断 Sleep、Join、Wait 方法造成的等待过程，默认将会抛出异常]{.red}
    • [Interrupt 方法配合 IsInterrupted \ Interrupted 方法可以实现优雅地停止线程]{.red}
    • [Sleep、Join、Wait 方法在被中断之后默认将中断标志重新设置为 false]{.pink}

IsInterrupted 方法

  • 作用：[获取线程的中断标志位的状态]{.red}

  • 细节：

    • [IsInterrupted 方法 默认 不会清除中断标志位，但是可以设置清除中断标志位]{.pink}

    • [IsInterrupted 方法获取的 调用该方法的线程对象 的中断标志位状态，而不是调用线程的]{.red}

  • 使用

    • 核心方法：thread.isInterrupted()（实例方法）

    • 代码演示：[Interrupt + IsInterrupted 实现优雅的停止线程]{.red}

Interrupted 方法

  • 作用：[获取线程的中断标志位状态，并且重新将中断标志位置为 false]{.red}

  • 细节：

    • [返回的结果是被 重置之前 的结果，而不是重置之后的结果]{.pink}

    • [IsInterrupted 方法获取的是 调用线程 的状态，而不是其他线程对象的状态]{.red}

  • 使用：

    • 核心方法：Thread.isInterrupted()（静态方法）
